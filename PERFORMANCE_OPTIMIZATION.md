# 性能优化方案

## 目标

这份文档用于约束后续性能优化方向。

后续如果继续做性能相关改动，优先按照本文档执行，避免反复回退到高开销实现。

## 当前结论

当前项目的性能瓶颈已经验证过几轮，结论如下：

1. 大量同类敌机的主要瓶颈不在追踪逻辑，而在显示对象数量和绘制方式。
2. `Graphics` 的逐帧重绘和频繁创建/销毁，是最应该优先避免的路径。
3. 同类对象共享纹理、对象池复用、减少场景树层级，是当前项目最有效的优化方向。
4. 性能优化应优先针对“压力测试路径”做，不要先为了通用性把实现复杂化。

## 当前已落地的优化策略

### 1. 敌机压力测试路径

当前压力测试敌机已经采用以下策略：

- 同类敌机共享纹理
- 每架敌机只保留单个 `Sprite`
- 不再走复杂的 `createShipScene()` 容器结构
- 命中框对象复用，不再频繁新建命中框对象

结论：

- 这是目前最核心、最有效的一层优化
- 后续不要轻易回退到“每架敌机使用多层 `Graphics` + `Container`”的方案

### 2. 子弹系统

当前玩家子弹和敌方子弹都已经采用：

- 共享纹理
- `Sprite` 复用
- 对象池回收

结论：

- 后续新增高频发射物时，默认参考子弹系统的做法
- 不要再用“每次开火 `new Graphics()`、命中后 `destroy()`”的方式实现

### 3. 追踪弹拖尾

当前追踪弹拖尾已经从矢量重绘改成：

- 固定历史点
- 固定数量的拖尾 `Sprite` 段
- 每帧只更新位置、旋转、长度、透明度

结论：

- 追踪弹拖尾后续应继续沿 `Sprite` 复用方向优化
- 不要回退到每帧 `Graphics.clear()` + 大量圆点/线条重画

### 4. 可关闭特效

当前设置中已经支持持久化开关：

- `FPS`
- `爆炸效果`

结论：

- 后续所有重型视觉特效，优先考虑是否需要提供开关
- 这样便于快速判断性能瓶颈归属

## 后续优化优先级

后续如果继续优化，优先级按下面顺序执行。

### 第一优先级

#### A. 继续压缩追踪弹目标选择成本

当前追踪弹仍有一项明确可优化点：

- `getNextTarget()` 还在做 `filter + sort`

后续应优先改成：

- 一次线性扫描
- 直接找最近或最合适目标

目标：

- 降低高密度目标场景下的 JS 计算成本

#### B. 继续减少高频对象分配

如果后续发现某个系统仍然频繁创建临时对象，优先改成：

- 复用数组
- 复用命中框
- 复用目标缓存
- 对象池

### 第二优先级

#### C. 爆炸/命中特效改成共享纹理 + 对象池

当前 `createImpactEffectSystem` 仍然属于高成本路径。

后续建议：

- 预生成 flash / spark 纹理
- 运行时用 `Sprite` 播放
- 用对象池复用特效实例

目标：

- 降低开火、命中、连锁爆炸时的帧率波动

### 第三优先级

#### D. 大量同类对象进一步改用 `ParticleContainer`

适用对象：

- 敌机压力测试精灵
- 子弹
- 简化拖尾段

使用前提：

- 只需要位置、缩放、旋转、透明度等有限属性
- 不依赖复杂嵌套结构

目标：

- 继续提升大规模同类精灵场景的批处理效率

## 后续实现注意事项

### 1. 不要优先优化错误的层

如果出现掉帧，先区分：

- 静止时掉帧
- 开火时掉帧
- 命中/爆炸时掉帧
- 追踪弹大量出现时掉帧

先定位，再下手，不要一上来同时改多个系统。

### 2. 不要轻易回退到 `Graphics` 逐帧重绘

以下做法默认视为高风险实现：

- 每帧 `clear()`
- 每帧画很多 `circle()`
- 每帧大量 `stroke()`
- 高频 `new Graphics()`
- 高频 `destroy()`

如果必须用 `Graphics`，要先证明：

- 数量很少
- 生命周期不频繁
- 没有更合适的共享纹理实现

### 3. 不要为了“看起来灵活”牺牲压力测试性能

压力测试场景优先级：

1. 稳定压测
2. 性能结论清晰
3. 实现简洁
4. 再考虑抽象复用

如果某个实现同时满足“更通用”但“更慢”，压力测试路径不应优先采用。

### 4. 先做数据结构优化，再做视觉微调

顺序应优先是：

1. 共享纹理
2. 对象池
3. 缓存数组/命中框
4. 线性扫描替代排序
5. 再做拖尾、颜色、层次等视觉微调

不要反过来。

### 5. 每次只验证一个变量

后续测试建议一次只改一个维度，例如：

- 只关爆炸效果
- 只减少追踪弹
- 只增加敌机数量
- 只替换某个系统为对象池版本

否则难以判断收益来源。

## 推荐的后续执行规则

后续如果用户继续提性能问题，默认按这个顺序处理：

1. 先确认掉帧发生阶段
2. 优先检查是否存在 `Graphics` 高频重绘
3. 优先改为共享纹理 + `Sprite`
4. 再补对象池
5. 再处理排序、目标查找、碰撞缓存
6. 最后才调视觉参数

## 不建议的回退项

后续优化中，以下内容默认不要回退：

- 敌机共享纹理 `Sprite` 实现
- 子弹共享纹理 + 对象池
- 追踪拖尾的 `Sprite` 段实现
- 设置中的性能调试开关
- 命中框缓存复用

如果必须回退，先说明原因并明确收益是否大于性能损失。
